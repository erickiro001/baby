## 搭建后端框架
- 技术栈: Go 1.26 + Gin + GORM + glebarez/sqlite (纯Go, 无CGO) + golang-jwt + bcrypt + Air 热更新
- 项目结构: `cmd/server/main.go` 入口, `internal/` 下按 config / database / models / services / handlers / middleware / utils 分层
- User 模型: ID, Username, Email, PasswordHash, CreatedAt, UpdatedAt, DeletedAt (软删除), PasswordHash 通过 `json:"-"` 禁止序列化返回
- 认证: 注册(用户名+邮箱唯一性校验, bcrypt cost=12), 登录(支持用户名或邮箱), JWT Bearer Token
- 安全中间件: JWT Auth (Bearer Token解析), CORS, SecurityHeaders (XSS/CSP/Frame), NoCache, SizeLimit (1MB), RateLimiter (Token Bucket, 10req/min per IP)
- API 路由: `/api/health`, `/api/v1/auth/register`, `/api/v1/auth/login` (公开+限流), `/api/v1/profile` (需JWT)
- 统一响应格式: {code, message, data}
- 配置: 通过 .env 文件 + 环境变量, 默认端口 8080, JWT 过期 24h
- Air 热更新已配置, Makefile 提供 dev/build/run/clean 命令
- 编译验证通过, 生成 35MB 可执行文件
- 服务已启动运行在 localhost:8080

## 中间件完备性 — 已补全
全局链顺序: RequestID → Logger → Recovery → RequestTimeout(30s) → CORS → SecurityHeaders → HSTS(dev=0) → CSP(strict) → NoCache → SizeLimit(1MB)
- RequestID: 每个请求生成 UUID，响应头 X-Request-ID + gin context 注入，客户端传的优先复用
- RequestTimeout: 30 秒请求截止，超时返回 504
- HSTS: 仅在 TLS 下生效，开发环境 maxAge=0 禁用
- CSP: 严格策略 `default-src 'none'; frame-ancestors 'none'`，API 不需要宽松策略
- CORS 仍为全放通 `*`，生产需收紧

## 前端 API 接口使用审计
- 全面审计 front/src/ 下所有 API 调用，生成报告 front/API_AUDIT.md
- **核心发现：前后端数据断层严重**
  - 前端 9 个 API 模块共 34 个函数已完整定义，但仅 4 个被实际调用（11.8%）
  - store 中只 import 了 timelineApi 和 babyApi 两个模块
  - family / works / milestones / health / capsules 5 个模块完全未被任何文件引用（死代码）
  
- **实际被调用的接口**：
  1. POST /api/v1/auth/login (LoginPage 直接调用)
  2. GET /api/v1/profile (App.tsx 启动验证 token)
  3. GET /api/v1/babies (store.fetchBabies)
  4. GET /api/v1/timeline (store.fetchTimeline)

- **所有增删改操作全走本地 zustand state**，不经过后端：
  - toggleLike、addComment、deleteTimelineEntry、toggleFeatured 等只改内存
  - 刷新页面后所有用户操作丢失
  
- **Demo 数据隐蔽问题**：initialState 预置硬编码 demo 数据，fetchInitialData 失败时静默回退，用户无法区分真实/演示数据

- **建议**：P0 补齐读操作初始数据拉取，P1 所有写操作改调后端 API + 乐观更新，P2 添加注册入口和发布功能入口
